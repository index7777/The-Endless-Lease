# 《無期租寓》任務記錄與完成驗收最高規則 v1.0

最後更新：2026-07-16
規則層級：所有長任務、多檔案修改、測試、素材生成與跨裝置工作的最高任務追蹤規則。

## 1. 目的

Codex 不得只靠聊天上下文記住任務。每項使用者要求必須先寫入專案任務帳本，之後才能實作。任何未記錄、未驗收或未回報的項目都不能標記完成。

## 2. 收到要求後的固定動作

1. 將要求拆成獨立、可驗收的項目。
2. 寫入本次任務紀錄，並保留使用者要求的完整原意。
3. 標註修改範圍與禁止修改範圍。
4. 為每項建立完成條件、測試與驗收證據要求。
5. 回讀任務帳本並向製作人重述清單。
6. 將任務狀態由 `PLANNING` 改為 `IN_PROGRESS` 後才開始修改。

禁止直接開始改檔案。

## 3. 固定任務檔案

```text
tasks/
├── ACTIVE_TASK.md
├── TASK_HISTORY.md
└── sessions/
```

- `ACTIVE_TASK.md`：只保存目前執行中的唯一正式清單。
- `TASK_HISTORY.md`：保存完成、取消、延後與失敗摘要。
- `tasks/sessions/`：保存每次任務完整封存，不得覆蓋舊紀錄。

## 4. ACTIVE TASK 必要內容

必須包含：任務名稱、時間、Git 分支、起始 Commit、來源、總狀態、使用者原始要求、Codex 拆解、理解、衝突、必做項目、逐項驗收、本次禁止修改、風險、新發現與工作日誌。

使用者原始要求與 Codex 拆解必須同時保存；不得只留下模糊摘要。

## 5. 任務拆解

每項只描述一個可以獨立判斷的結果。禁止「改善角色動畫」等不可驗收描述。應拆為比例、腳底誤差、步態階段、亮度差與攻擊階段等獨立數值或畫面結果。

## 6. 狀態流程

項目只能使用：

```text
TODO
IN_PROGRESS
BLOCKED
NEEDS_REVIEW
VERIFIED
CANCELLED
```

正確流程為 `TODO → IN_PROGRESS → NEEDS_REVIEW → VERIFIED`。程式已寫不等於完成；只有通過驗收且有證據才能 `VERIFIED`。

取消需求不得刪除舊項目；標記 `CANCELLED`、保留原要求與原因，並指向替代項目。

## 7. 強制回讀時機

至少在以下時機完整回讀 `ACTIVE_TASK.md`：

- 開始修改前
- 完成第一個檔案後
- 每完成兩個項目後
- 執行測試前
- 測試失敗、準備修正前
- 產生或替換素材後
- 準備宣告完成前

長任務不得依記憶工作。

## 8. 工作日誌

只記錄重要決策、修改、量測與測試，不需記錄每個命令。程式已完成但尚未通過解析度、視覺或實機驗收時，必須明確記為 `NEEDS_REVIEW`。

## 9. 新發現與追加需求

發現清單外問題時先記入「新發現但不在本次範圍」：

- 不阻擋：只記錄，不修改。
- 阻擋：標記 BLOCKED 並說明。
- 需要擴大範圍：先更新帳本再修改。
- 會改玩法或定案：停止並詢問製作人。

使用者追加要求時，必須暫停、加入下一個 T 編號、記錄追加時間、更新影響與驗收後再繼續。不得只在聊天回覆收到。

## 10. 任務編號追溯

重要修改應在提交訊息、測試或工作紀錄中對應 T 編號，例如：

```text
fix(animation): align player foot anchors [T02]
feat(animation): complete scale and motion validation [T01-T05]
```

不要求在正式程式碼塞入大量任務註解。

## 11. 完成前強制核對

宣告完成前，逐項確認是否已修改、已測試、有驗收證據且無回退。任何必做項仍為 `TODO`、`IN_PROGRESS`、`BLOCKED` 或 `NEEDS_REVIEW` 時，不得宣告整體完成。

每項 `VERIFIED` 至少必須有：自動測試、截圖路徑、數值量測、Contact Sheet、視覺差異報告、實際操作結果或建置輸出之一。「看起來正常」不算證據。

## 12. 完成報告

最終回報必須包含：

- 已驗證完成
- 部分完成及尚缺內容
- 阻擋與需要的外部條件
- 未處理新發現
- Build、自動測試、視覺回歸、手機橫向、Placeholder 檢查
- 修改檔案
- 驗收素材
- Git Commit 與 Push 狀態

禁止只回覆「已修改完成，請查看」。

## 13. 封存與跨裝置續作

任務結束時將總狀態改為 `COMPLETED`、`PARTIAL` 或 `BLOCKED`，複製到 `tasks/sessions/`，更新 `TASK_HISTORY.md`，再清空或建立下一份 ACTIVE TASK。封存檔與程式修改必須進入同一 Git 版本。

新對話或另一台電腦先讀 ACTIVE TASK、TASK HISTORY、待辦與總交接。ACTIVE TASK 有未驗證項目時先續做；聊天衝突時依製作人最新明確指示更新帳本，不得自行重做歷史聊天要求。

## 14. 自我檢查

進入下一階段前必須回答：

- 是否已寫入全部要求？
- 是否仍有要求只存在聊天？
- 是否修改了清單外內容？
- 是否把已寫程式誤認完成？
- 每項完成是否有證據？
- 是否仍有被遺忘項目？

任何答案不確定就重新回讀帳本。

## 15. 自動檢查

- `npm run task:check`：檢查 ACTIVE TASK 結構、未完成項、VERIFIED 證據、Placeholder、工作日誌與起始 Commit。
- `npm run precommit:project`：先執行任務檢查，再執行完整測試；有未驗證必做項時必須阻擋提交前完成判定。
