# AGENTS.md

## 專案目的

本專案中的骰子動畫必須達到可直接用於正式遊戲的品質。

Codex 必須獨立完成以下工作：

* 分析現有動畫問題
* 判斷應修復或重建
* 建立穩定的骰子動畫
* 輸出遊戲可用素材
* 建立自動檢查工具
* 在目標遊戲引擎中驗證
* 提供完整的驗證報告

「已產生圖片」不代表工作完成。

只有當骰子動畫在遊戲中播放流暢、旋轉合理、外觀穩定、結果正確，而且不需要人工逐幀修圖或重新對齊時，才可標記為完成。

---

# 1. 核心開發原則

## 1.1 骰子是剛體，不是逐幀插畫

骰子必須被視為單一且不可變形的剛體物件。

所有動畫幀必須來自同一個骰子幾何結構、同一套骰面配置、同一套材質和同一套攝影機設定。

禁止將每一幀當成獨立插畫自由生成。

禁止在不同幀中改變：

* 骰子尺寸
* 長寬比例
* 邊角形狀
* 倒角尺寸
* 骰面點數排列
* 點數大小與間距
* 材質
* 顏色
* 線條粗細
* 透視規則
* 攝影機焦距
* 光源方向
* 畫布尺寸
* 像素密度

動畫中只允許連續改變：

* 世界座標位置
* 旋轉角度
* 移動速度
* 旋轉速度
* 接觸狀態
* 反彈狀態
* 陰影位置與形狀，但必須符合固定光源和物理位置

---

## 1.2 優先使用可控制的動畫系統

骰子動畫應優先使用以下方式建立：

1. 3D 骰子模型與骨架無關的剛體動畫
2. Blender 關鍵幀動畫
3. Blender 剛體物理模擬
4. Unity 或 Godot 的剛體模擬
5. 使用程式計算位置與四元數旋轉
6. 從固定攝影機批次渲染 PNG 序列或 sprite sheet

只有在無法使用可控制幾何模型時，才允許修復既有 PNG 幀。

若既有 PNG 幀存在明顯的形變、點數漂移、透視錯誤或旋轉不連續，Codex 不得繼續用圖片修補掩蓋問題，必須改用單一骰子模型重建整套動畫。

---

## 1.3 圖片生成只能負責外觀

圖片生成工具只能用於：

* 建立骰子材質
* 建立表面風格
* 建立參考圖
* 修復輕微透明邊緣
* 修復不影響幾何結構的局部視覺瑕疵

圖片生成工具不得自行決定：

* 骰子幾何形狀
* 骰面朝向
* 每幀旋轉角度
* 骰子的物理路徑
* 落地反彈
* 最終停面
* 每幀攝影機角度

位置、旋轉、碰撞和停止結果必須由程式、時間軸或物理模擬控制。

---

# 2. 骰面規則

## 2.1 標準骰面配置

若使用標準六面骰，必須遵守以下對面關係：

* 1 的對面是 6
* 2 的對面是 5
* 3 的對面是 4

同一顆骰子的骰面配置必須從動畫第一幀到最後一幀完全一致。

禁止出現以下情況：

* 同一個點數出現在兩個不同方向
* 對面關係在動畫中改變
* 骰面圖案隨旋轉重新排列
* 點數在骰面上滑動
* 點數大小、距離或方向逐幀變化
* 最終顯示結果與實際模型朝向不一致

## 2.2 骰面配置檔

Codex 應建立可程式化讀取的骰面設定，例如：

```json
{
  "top": 1,
  "bottom": 6,
  "front": 2,
  "back": 5,
  "right": 3,
  "left": 4
}
```

實際配置可以不同，但必須符合標準對面關係，並在整段動畫中保持不變。

# 3. 動畫規格

每套骰子動畫開始製作前，Codex 必須先建立動畫規格檔。

建議位置：

```text
animation/dice_roll/animation_spec.json
```

最低必要欄位如下：

```json
{
  "name": "dice_roll",
  "frame_count": 24,
  "fps": 24,
  "loop": false,
  "canvas_size": [128, 128],
  "cell_size": [128, 128],
  "pivot": [64, 104],
  "baseline_y": 104,
  "camera_type": "orthographic",
  "camera_locked": true,
  "lighting_locked": true,
  "transparent_background": true,
  "target_top_face": 6,
  "motion_phases": [
    "entry",
    "air_rotation",
    "first_contact",
    "bounce",
    "secondary_rotation",
    "friction_slowdown",
    "settle"
  ]
}
```

所有輸出工具、驗證工具和遊戲引擎匯入工具都必須以此規格檔為主要資料來源，禁止在不同腳本中重複寫死互相矛盾的數值。

# 4. 動畫運動要求

## 4.1 動畫階段

骰子投擲或翻滾動畫至少必須包含：

1. 進場或起始運動
2. 空中旋轉
3. 第一次接觸地面
4. 反彈或翻面
5. 次要旋轉
6. 摩擦減速
7. 最終穩定停止

不得只使用固定速度旋轉後突然停止。

不得讓骰子在最後一幀瞬間吸附到指定結果。

## 4.2 位移曲線

骰子的位移必須符合合理的速度變化：

* 空中階段應有重力影響
* 落地前速度應合理增加
* 碰撞後垂直速度應反向並衰減
* 水平速度應因摩擦逐漸降低
* 最終停止前不應出現小幅度無限滑動
* 不得在沒有碰撞原因的情況下突然改變方向

## 4.3 旋轉曲線

旋轉必須使用連續的旋轉資料。

建議使用：

* Quaternion
* Slerp
* 穩定的 Euler 曲線與角度展開
* 剛體角速度

禁止直接對未處理的 Euler 角度做可能跨越 `180°/-180°` 的錯誤線性插值。

禁止出現：

* 旋轉方向突然反轉
* 單幀角度跳變
* 骰面瞬間切換
* 旋轉軸無原因改變
* 骰子停止後仍有視覺抖動
* 最終幾幀反覆在兩個角度之間切換

## 4.4 接觸與反彈

骰子接觸地面時必須符合以下要求：

* 接觸點不能穿過地面
* 接觸前後的速度方向必須合理
* 反彈高度必須低於前一次運動高度
* 每次反彈的能量必須逐漸降低
* 接觸時可有快速角速度改變，但不得造成形體變形
* 最終停止時至少應有一個合理穩定的支撐面

# 5. 攝影機、光源與陰影

## 5.1 攝影機

整套動畫必須使用固定攝影機。

除非動畫規格明確要求，否則禁止：

* 攝影機移動
* 攝影機縮放
* 攝影機旋轉
* 焦距改變
* 透視強度改變
* 每幀自動重新構圖

若輸出為 2D sprite，建議使用固定正交攝影機，以降低透視漂移。

## 5.2 光源

光源方向、強度、顏色和陰影設定必須鎖定。

骰子旋轉時，表面明暗可以因法線方向而改變，但光源本身不得逐幀改變。

禁止出現：

* 同一骰面在相近角度下亮度突然跳變
* 高光位置不符合骰子旋轉
* 無原因改變光源方向
* 每幀產生不同風格的陰影
* 骰子已停止但光影仍抖動

## 5.3 陰影

陰影必須由固定光源、地面和骰子位置共同決定。

陰影需要符合：

* 骰子越靠近地面，陰影通常越集中
* 骰子離地越高，陰影可適度變淡或擴散
* 陰影中心應與骰子位置具有合理關係
* 骰子停止後，陰影必須完全穩定
* 不得將陰影直接畫死在骰子 PNG 上，除非專案規格明確要求預渲染陰影

---

# 6. PNG 與 Sprite Sheet 規則

## 6.1 固定畫布

所有動畫幀必須使用完全相同的：

* 畫布尺寸
* 格子尺寸
* pivot
* baseline
* 像素密度
* 色彩模式
* Alpha 格式

禁止逐幀自動裁切。

禁止逐幀自動置中。

禁止依每幀內容改變 bounding box 後重新縮放。

即使骰子在某一幀只占畫面的一小部分，也必須保留固定畫布。

## 6.2 透明背景

透明背景必須乾淨。

禁止出現：

* 白色或黑色邊框
* 半透明雜點
* 前一幀殘影
* Alpha 邊緣閃爍
* 不一致的預乘 Alpha
* 色彩空間轉換造成的邊緣污染

## 6.3 像素風規則

若專案為像素風：

* 必須使用固定像素解析度
* 必須使用固定色盤
* 必須使用 nearest-neighbor 縮放
* 禁止雙線性或雙三次插值
* 禁止每幀使用不同的像素化結果
* 禁止出現半像素位置
* 物件移動需依專案規則選擇整數像素或受控的次像素渲染

---

# 7. 既有動畫修復流程

當任務是修復既有骰子 PNG 動畫時，Codex 必須依照以下流程執行。

## 7.1 輸入分析

先讀取所有動畫幀並取得：

* 幀數
* FPS
* 畫布尺寸
* Alpha 範圍
* 骰子 bounding box
* 骰子中心點
* 視覺質心
* 主要角點
* 輪廓遮罩
* 接地點
* 陰影中心與面積
* 相鄰幀差異
* 可能的骰面朝向
* 旋轉方向
* 位移速度
* 角速度趨勢

必須先輸出問題報告，再開始修正。

## 7.2 問題分類

問題至少分類為：

* 畫布不一致
* 裁切不一致
* 中心點抖動
* 尺寸漂移
* 骰子形變
* 透視漂移
* 骰面錯誤
* 旋轉跳幀
* 旋轉方向反轉
* 位移速度不連續
* 接地點錯誤
* 穿透地面
* 反彈不自然
* 陰影跳動
* Alpha 邊緣問題
* 最終停面錯誤
* FPS 或播放時間設定錯誤

## 7.3 修復或重建判斷

只有符合以下條件時，才可修復現有 PNG：

* 骰子幾何大致一致
* 骰面配置可辨識且一致
* 主要透視沒有嚴重改變
* 缺陷主要來自對齊、時間或少量異常幀
* 相鄰幀仍可推斷合理運動

出現以下任一情況時，必須重建：

* 多數幀形狀不同
* 骰面點數頻繁改變
* 攝影機角度不一致
* 光源方向逐幀漂移
* 旋轉順序無法推斷
* 大量中間姿勢缺失
* 最終結果與前面旋轉無法銜接
* 修補成本高於重建
* 無法保證正式遊戲品質

Codex 不得為了保留錯誤素材而選擇較差的技術方案。

---

# 8. 修復方法

## 8.1 可接受的修復

允許的修復方式包括：

* 固定畫布重新輸出
* 依固定 pivot 對齊
* 修正 baseline
* 平滑中心點路徑
* 平滑速度曲線
* 修正旋轉角度
* 使用 Quaternion 插值補中間幀
* 以固定模型重新渲染異常幀
* 清除透明邊緣雜點
* 統一色彩與 Alpha
* 重新計算陰影
* 重新設定正確 FPS
* 補足缺失的過渡幀

## 8.2 禁止的假修復

禁止以下方式：

* 將明顯錯誤幀直接刪除而不修正時間
* 使用模糊效果掩蓋跳幀
* 只增加 motion blur 掩蓋旋轉錯誤
* 逐幀重新生成不同骰子
* 單純複製前後幀填補缺失幀
* 強制將每幀縮放到相同 bounding box
* 讓陰影獨立於骰子運動
* 在最後一幀直接替換成指定點數圖片
* 未驗證就宣告修復完成

Motion blur 只能作為合理速度下的視覺效果，不能代替正確的旋轉和位移。

# 9. 重建流程

若判斷需要重建，Codex 必須：

1. 建立單一骰子模型
2. 建立固定骰面配置
3. 設定固定材質
4. 設定固定攝影機
5. 設定固定光源與地面
6. 指定目標 FPS、幀數與持續時間
7. 設定目標最終停面
8. 建立或模擬合理的投擲運動
9. 將物理結果烘焙為穩定關鍵幀
10. 修正不必要的小幅物理抖動
11. 使用固定設定批次渲染
12. 產生 PNG 序列
13. 產生 sprite sheet
14. 產生預覽 GIF 或 MP4
15. 匯入目標遊戲引擎測試
16. 執行全部自動驗證

若使用隨機物理模擬，必須保存 random seed，以便重現完全相同的動畫。

若動畫結果需要固定，例如最終必須顯示 6，必須確保實際旋轉和最終模型朝向都正確，不得只替換最後顯示圖片。

```text
tools/validate_dice_faces.py
tools/validate_motion_continuity.py
tools/validate_alpha_edges.py
```

```text
tools/blender/render_dice_frames.py
```

工具必須：

* 可由命令列執行
* 參數清楚
* 有錯誤處理
* 不覆蓋來源檔
* 將輸出放在明確目錄
* 使用動畫規格檔
* 產生可讀取的報告
* 在失敗時回傳非零 exit code
* 可在相同輸入下重現相同結果

# 11. 自動驗證標準

動畫完成後，必須執行自動驗證。

## 11.1 檔案驗證

檢查：

* 幀數是否正確
* 檔名順序是否正確
* 所有畫布尺寸是否一致
* 色彩模式是否一致
* Alpha 格式是否一致
* 是否有缺幀
* 是否有重複幀
* sprite sheet 格線是否正確
* 預覽播放速度是否符合 FPS

## 11.2 幾何穩定性

檢查：

* 骰子長寬比例是否穩定
* 輪廓是否出現不合理形變
* 邊角尺寸是否突然改變
* 骰面點數大小是否穩定
* 點數位置是否固定在骰面局部座標
* 骰子縮放是否符合實際景深或固定正交攝影機
* 停止後的最後數幀是否完全穩定

## 11.3 運動連續性

檢查相鄰幀的：

* 中心點位移
* 速度
* 加速度
* 旋轉角度
* 角速度
* 角加速度
* 接地點
* 反彈高度
* 滑動距離

必須標示異常尖峰。

不得只以單一平均值判定，必須找出具體異常幀編號。

## 11.4 骰面正確性

檢查：

* 所有骰面是否符合固定配置
* 對面關係是否正確
* 可見骰面是否符合目前旋轉
* 點數方向是否合理
* 最終朝上點數是否符合規格
* 最終結果是否與遊戲邏輯一致

## 11.5 地面與陰影

檢查：

* 骰子是否穿透地面
* 接觸幀是否有合理支撐點
* 停止後是否漂浮
* 陰影中心是否合理
* 陰影面積變化是否平順
* 光源方向是否一致
* 陰影是否在骰子停止後保持穩定

## 11.6 視覺驗證

必須輸出以下預覽：

* 正常速度預覽
* 慢速預覽
* 逐幀編號預覽
* 骰子中心點軌跡
* 接地點軌跡
* bounding box 疊圖
* 相鄰幀差異圖
* 修正前後並排比較

---

# 12. 建議誤差限制

實際數值可依素材解析度調整，但必須記錄於動畫規格。

預設可採用：

```json
{
  "max_still_center_jitter_px": 0.5,
  "max_unexplained_scale_change_ratio": 0.02,
  "max_shape_area_change_ratio": 0.05,
  "max_baseline_penetration_px": 1,
  "max_final_motion_px": 0.25,
  "max_final_rotation_degrees": 0.25,
  "duplicate_frame_threshold": 0.995,
  "alpha_noise_threshold": 0.01
}
```

這些限制不可機械套用。

例如骰子接近或遠離透視攝影機時，尺寸可以合理變化；但該變化必須符合攝影機與位置，而不是無規則漂移。

---

# 13. 目標遊戲引擎驗證

動畫不能只在 GIF 或圖片檢視器中驗證。

Codex 必須在目標遊戲引擎建立最小測試場景，至少確認：

* 正確載入全部幀
* 正確使用目標 FPS
* pivot 正確
* baseline 正確
* 循環設定正確
* 非循環動畫能停在正確最終幀
* 播放期間不會縮放或位移跳動
* 透明背景正確
* 材質不出現白邊或黑邊
* UI 模式與世界物件模式皆符合專案需求
* 骰子結果與遊戲邏輯同步
* 不同解析度與縮放倍率下表現正常

如果骰子動畫是由遊戲引擎即時控制，應測試不同幀率下的結果，避免動畫與實際 FPS 綁死。

# 14. 輸出目錄規範

建議使用以下結構：


# 17. 失敗處理原則

發現問題時，Codex 應先定位根本原因。

禁止只處理表面症狀。

例如：

* 中心點抖動可能來自逐幀裁切，不應只手動移動幀
* 骰面跳動可能來自獨立圖片生成，不應只重畫點數
* 旋轉突然反向可能來自 Euler 角度跨界，不應只刪除異常幀
* 最終停面錯誤可能來自模型配置，不應只替換最後一張圖片
* 陰影跳動可能來自光源或地面設定，不應逐幀手動畫影子

修正單一幀後，必須重新執行整套驗證，確認沒有破壞相鄰幀和最終結果。

# 18. 完成定義

只有同時符合以下條件，骰子動畫才算完成：

* 骰子全程保持同一幾何形狀
* 骰面配置全程一致
* 位移連續且符合運動邏輯
* 旋轉連續且沒有角度跳變
* 接觸、反彈、摩擦和停止合理
* 攝影機固定
* 光源固定
* 陰影穩定
* 畫布、pivot 和 baseline 一致
* PNG 邊緣乾淨
* sprite sheet 可正確讀取
* 最終停面符合指定結果
* 自動驗證全部通過
* 遊戲引擎測試通過
* 無須人工逐幀修圖或重新對齊
* 所有來源與工具足以重現輸出

最終判定原則：

> 不要把骰子動畫當成一組分開的圖片。必須把它當成同一顆剛體骰子在連續時間中的位置、旋轉、碰撞與減速結果。任何無法用固定幾何和連續運動解釋的幀，都是需要修正的異常幀。
