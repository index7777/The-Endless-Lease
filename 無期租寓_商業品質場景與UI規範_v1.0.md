# 商業品質場景呈現、角色動畫、物品與 UI 開發規範

## 1. 專案目標

本專案必須以「可正式販售的商業遊戲」為最低完成標準。

Codex 應獨立完成場景分析、系統設計、實作、素材整理、除錯、跨場景驗證、效能測試及最終品質檢查，不得只針對單張畫面加入臨時補丁。

所有視覺、動畫與 UI 系統必須共同服務以下目標：

- 玩家能立即理解角色、場景、出口、可互動物及當前危險。
- 場景具有明確光影層次、時間感與空間深度。
- 角色看起來真正存在於環境內，而不是貼在背景圖片上。
- 地面物件具有重量、透視、接觸與可辨識性。
- UI 清楚但不搶走場景注意力。
- 所有畫面在不同解析度、幀率、場景與時間狀態下保持一致。
- 系統可以重複套用到所有樓層、房間、走廊與管理室。

不得以「目前能運作」作為完成標準。

完成標準必須是：

> 遊戲在實際遊玩時具有穩定的視覺層級、可信的場景光照、自然角色動畫、清楚互動反饋及一致 UI，且不需要逐場景人工補位或修改。

---

# 2. 禁止使用的開發方式

禁止以下做法：

- 為單一場景手動覆蓋整張色調，卻沒有正式光照系統。
- 只在畫面上加一層黑色或黃色透明色表示晚上或白天。
- 每個房間各自寫死燈光、角色亮度及物品偏移。
- 只用角色腳底陰影掩蓋角色與場景不貼合。
- 讓地面物品持續上下漂浮來表示可拾取。
- 將所有互動物永久加上高亮外框。
- 將大量狀態永遠顯示在四個畫面角落。
- 為修正單一動畫而改變整個角色根節點。
- 用 motion blur、畫面震動或粒子掩蓋動畫不連續。
- 使用過小字體塞入大量說明。
- 將 UI 數值直接寫死在畫面程式中。
- 未進行實際遊戲測試就標記完成。
- 只在 1080p、單一 FPS 或單一房間測試。
- 為了快速完成而產生需要人工逐張修正的素材。

所有修正都必須回到共用系統、資料設定或標準素材流程中。

---

# 3. 畫面視覺層級

每個遊戲場景必須建立明確的四層視覺層級：

1. 玩家角色與立即危險
2. 可互動物與任務目標
3. 場景結構與可行走區域
4. 純氛圍裝飾與背景資訊

不得讓每一個物件都具有相同亮度、對比與銳利度。

角色周圍應有足夠輪廓辨識度，但不得像被獨立打了一盞舞台燈。

重要出口、門牌與互動點可以使用：

- 局部光源
- 材質反光
- 微弱色溫差
- 構圖引導
- 靠近時才出現的互動提示

不重要的背景區域可以降低：

- 亮度
- 對比
- 飽和度
- 細節清晰度
- 動態頻率

玩家進入場景後，應在五秒內辨認：

- 自己的位置
- 可以前往的方向
- 最近的可互動物
- 當前任務或威脅
- 時段與整體狀態

# 4. 燈光與氛圍系統

## 4.1 不得只使用均勻環境光

目前類似走廊場景不應整體使用相同亮度。

每個場景至少應有：

- 基礎環境光
- 主要功能光源
- 局部引導光
- 接觸陰影
- 遮蔽或暗部
- 必要的反射或地面亮斑
- 角色與物品的環境光響應

例如走廊可使用：

- 頂部日光燈作為冷色主光
- 房門附近的小燈作為暖色引導光
- 遠處較暗或偶發閃爍的區域形成危險感
- 潮濕地面產生受控反光
- 門框、牆角、角色腳底有接觸陰影

禁止將所有黑位壓死。

玩家必須仍能閱讀：

- 角色輪廓
- 地板邊界
- 房門位置
- 可拾取物
- UI 文字

## 4.2 光源必須影響環境

燈具不能只是背景圖片中的亮色區域。

每盞可見燈具都應有合理的視覺影響：

- 對附近牆面及地面的亮度影響
- 對角色的明暗或色溫影響
- 對物品的高光與陰影影響
- 對地面反射的影響
- 必要的柔和陰影
- 開關或故障時的場景反應

若專案使用預渲染背景，Codex 必須建立以下其中一種可重複方案：

- 場景光照遮罩
- 區域式角色色調與亮度設定
- 2D light map
- 角色 normal map 與 2D 光源
- 多組時段背景及平滑轉換
- 光照探針區域或自訂環境取樣

禁止角色永遠使用同一套亮度與色彩。

## 4.3 光照區域

每個場景應定義可查詢的光照區域：

```text
LightingZone
├── ambient_color
├── ambient_intensity
├── character_tint
├── character_exposure
├── shadow_strength
├── reflection_strength
├── fog_or_haze
└── audio_profile

角色和地面物品進入不同區域時，光照設定應平滑過渡，不得瞬間跳色。

參考過渡時間：

0.25 至 0.8 秒

實際值應依移動速度與場景風格調整。

5. 時段表現
5.1 時間不能只顯示在 HUD

若 HUD 顯示 第 1 日 10:07，世界畫面也必須能感受到相應時段。

時間狀態至少可分為：

清晨
白天
傍晚
夜晚
深夜
特殊異常時段

室內場景不一定依靠陽光表現時間，可透過以下方式建立差異：

日光燈亮度及穩定度
窗縫或通風口的環境光
走廊遠端色溫
管理室是否亮燈
房門燈與警示燈狀態
住戶活動聲音
電梯運作頻率
場景 NPC 或影子
環境音
牆上時鐘
管線聲、水聲與樓層廣播
可互動物狀態

例如：

白天
主燈較穩定
冷灰色環境光稍高
走廊較容易辨識
住戶活動聲較多
陰影較柔和
傍晚
遠處出現較暖色燈光
亮暗差增加
管理室開始亮燈
走廊反射更明顯
夜晚
部分燈具關閉
角色輪廓更依賴局部光源
房門縫、樓梯口及緊急燈成為引導
環境音減少並增加低頻噪音
深夜
不應只是整體變黑
應改變可用燈具、聲音、事件與危險配置
部分房門燈或門牌可能不正常
管理室及電梯有特殊狀態
異常事件生成規則改變
5.2 時間系統

Codex 必須建立統一遊戲時間服務：

GameTimeService
├── current_day
├── current_hour
├── current_minute
├── time_period
├── time_scale
├── paused
└── scheduled_events

時段變更必須發送事件：

OnMinuteChanged
OnHourChanged
OnTimePeriodChanged
OnDayChanged

燈光、環境音、NPC、異常事件與 UI 必須訂閱同一個時間來源。

禁止各系統自行讀取或計算不同時間。

時段轉換必須平滑，不得在某一分鐘整個場景突然跳色。

6. 角色與場景的光照整合

角色不能像獨立 PNG 貼在背景上。

Codex 必須檢查並統一：

角色光源方向
角色明暗對比
角色色溫
角色陰影方向
角色腳底接觸陰影
場景透視與角色比例
角色銳利度與背景銳利度
角色粒度、噪點與色彩空間

若場景為偏冷日光燈，角色不能仍使用暖黃攝影棚光。

若角色走到燈具正下方，應有合理亮度變化；走入暗處時應逐漸降低曝光，但仍保留必要輪廓。

角色與場景的色彩處理必須集中在 CharacterEnvironmentLighting 或等效系統，不得散落於個別場景程式。

7. 人物動畫商業品質規範
7.1 必要移動狀態

角色至少必須包含：

Idle
IdleVariant
WalkStart
WalkLoop
WalkStop
Turn
RunStart
RunLoop
RunStop
AttackPrepare
Attack
AttackRecover
Interact
PickUp
Push
Hurt
Knockback
Fall
Land
Dead

不得只用 Idle 和 Walk 處理全部行為。

7.2 起步、停止與轉身

角色開始移動時：

身體重心先轉移
前腳或後腳有合理起步
動畫與位移同步
不得從 idle 任意幀直接切入 walk loop

角色停止時：

應落到合理接觸幀
或播放 walk-stop
或使用受控 crossfade
不得在腳懸空時直接切回 idle

角色反向移動時：

先減速
播放轉身或短暫重心切換
再重新起步

禁止瞬間鏡像角色並保持原速度。

7.3 動畫速度與移動同步

必須校正：

實際移動距離
動畫循環時間
步幅距離
動畫播放倍率

基礎關係：

move_speed = stride_distance / cycle_duration

動畫播放倍率：

animation_speed_ratio =
actual_horizontal_speed / authored_horizontal_speed

比例超出合理範圍時，必須切換慢走、正常走路或跑步動畫，不能無限制加速同一段 walk。

7.4 腳步與重量

角色腳接觸地面時必須有：

腳底位置穩定
相應腳步音
必要的衣物與武器次要動態
與地板材質相符的反饋
適度的重心起伏
不得滑步或漂浮

角色世界根節點應穩定沿地面移動；自然的身體上下起伏只能發生在 VisualRoot。

7.5 Idle 動畫

Idle 不得完全靜止，也不得動作過多。

可包含：

呼吸
重心小幅轉移
查看走廊
握緊武器
調整肩帶
擦拭雨水或血跡
對附近聲音產生短暫反應

Idle variant 應低頻出現，不能每幾秒重複相同大動作。

7.6 動畫中斷

攻擊、拾取、互動、受擊與移動必須明確定義可中斷區間。

不得出現：

拾取過程中仍滑動
攻擊結束直接跳回完全不同姿勢
角色受擊時武器瞬移
動畫切換造成尺寸或腳底高度改變
互動結束後角色朝向錯誤
8. 地面物品與拾取物

目前類似地面紙張若只用明亮圖片直接放在地板上，會像 UI 貼紙而不是世界物件。

每個拾取物必須使用統一 WorldPickup 系統：

WorldPickup
├── item_id
├── world_visual
├── contact_shadow
├── interaction_area
├── pickup_anchor
├── highlight_state
├── quantity
├── rarity_or_type
└── spawn_validation
8.1 地面貼合

所有物品生成時必須：

查詢地面位置。
對齊地面或預設表面。
套用合理透視和旋轉。
產生接觸陰影。
檢查不與牆壁、房門、角色及其他物件重疊。
檢查位於可到達區域。

紙張、票券等薄物件應：

貼近地面
有輕微不規則旋轉
邊緣可有捲曲或厚度
有方向一致的接觸陰影
亮度符合所在地區
不得像垂直面對攝影機的圖示

鑰匙、藥瓶等物件需有符合尺寸的高光、陰影及比例。

8.2 不要持續上下漂浮

偏寫實恐怖遊戲中，地面物品不應持續上下浮動。

可使用的可讀性提示：

靠近時出現微弱輪廓
短暫光澤掃過
微弱材質反射
一次性的紙角晃動
非持續性的環境風吹動
靠近才顯示名稱與按鍵提示
由燈光或構圖引導玩家注意

禁止：

所有物品永久發光
永久旋轉
高幅度上下浮動
永久顯示文字
使用與世界不一致的純白高亮
8.3 拾取互動

玩家進入互動範圍時才顯示：

E 拾取
物品名稱
數量
必要的用途摘要

拾取後應播放：

角色拾取或簡化互動動畫
物品移向角色／淡出／被手拿起
對應音效
簡短 UI 確認
背包或數值更新

不得點擊後物品瞬間消失且沒有任何反饋。

同一區域有多個物品時，必須提供最近目標、方向選擇或循環切換，不得隨機拾取。

9. UI 商業品質規範
9.1 UI 不得平均散落四角

UI 應分為三層：

永久必要 HUD

只保留玩家需要持續知道的資訊：

生命
體力或精神
當前裝備／快捷欄
關鍵租約數值
必要時間資訊
情境 HUD

只在需要時出現：

可互動提示
物品名稱
房門資訊
任務更新
狀態異常
危險警告
選單／檔案介面

在玩家主動開啟時顯示：

詳細住戶紀錄
完整租約狀態
任務細節
角色數值
物品清單
公寓狀態

目前右上角大量固定文字應移至可開啟的「住戶紀錄」或縮減為一個簡要狀態入口。

9.2 HUD 建議結構
左上

保留精簡時空資訊：

第 1 日
6F・601
10:07

當前租約只顯示一句最重要目標，例如：

前往 B1
剩餘 10:07

完整說明放入任務面板。

右上

只保留：

住戶紀錄入口
暫停／設定
極少量關鍵狀態

不要永久顯示所有完成數和系統狀態。

左下

生命、體力或精神應：

使用一致的尺寸與對齊
明確標示當前／最大值
只有狀態改變時提高亮度
平時降低存在感
危險時提供顏色、閃動或聲音反饋
下方中央

快捷欄需統一：

格子尺寸
數字位置
物品圖示尺寸
選取框
數量
可用／不可用狀態
鍵盤或手把提示

禁止使用低對比文字及無法辨識的灰色禁用狀態。

右下

租券數值可以作為遊戲核心貨幣保留，但應做成符合世界觀的票券、帳單或管理室印章式顯示，而不是普通數字方框。

9.3 UI 文字與尺寸

在 1080p 基準下：

主要 HUD 文字不得小於約 18 px 的可讀視覺尺寸。
次要提示不得小於約 16 px。
標題與主要數值應有清楚層級。
行距不得過密。
文字與背景必須有足夠對比。

數值、單位、標題、警告與說明必須使用固定文字樣式 token，不得每個介面自行設定字體大小。

9.4 統一設計系統

建立 UI design tokens：

UIColors
UIFontStyles
UISpacing
UIBorders
UIShadows
UIOpacity
UIAnimationDurations
UIInputIcons

例如：

color_text_primary
color_text_secondary
color_warning
color_danger
color_selected
color_disabled
color_panel_dark
color_paper

所有 UI 必須引用同一套 token。

禁止在各介面直接寫死顏色與尺寸。

9.5 UI 動畫

UI 動畫應低調且有功能性：

面板淡入：150–250 ms
任務更新：短暫高亮後回復
數值變化：舊值至新值平滑過渡
危險警告：有節制的閃動或呼吸效果
快捷欄切換：選取框滑動或淡入
互動提示：玩家靠近後柔和顯示

禁止所有元素同時移動。

禁止長時間阻擋操作的華麗轉場。

10. UI 與世界的沉浸整合

UI 應延續「無期租寓」的行政恐怖風格，但不可犧牲閱讀性。

可使用：

細金屬邊框
低透明深色底
舊紙標籤
印章式警告
微弱紅色異常提示
管理室表單語言
房號與登記編號系統

但戰鬥及即時 HUD 不應使用過厚紙張面板遮擋場景。

世界觀樣式與功能可讀性必須分離：

主選單、建檔、寄存櫃可使用較強紙張與檔案視覺。
即時遊戲 HUD 應更輕、更透明、更快速。
警告和互動提示必須比裝飾優先。
11. 環境音與燈光同步

商業品質氛圍不能只依靠畫面。

每個燈光與時段設定應配對環境音：

日光燈嗡鳴
閃爍前的電流聲
遠處電梯
水管聲
門後住戶聲
雨聲
腳步反射
管理室電話或印章聲
深夜降低一般活動聲
異常時段增加不明低頻或方向錯誤的聲源

燈光閃爍必須與音效及遊戲事件同步，不得隨機無意義閃爍。

12. 資料驅動架構

Codex 必須建立可重複配置的資料資產，而不是把所有規則寫在場景程式裡。

建議資料：

LightingProfile
TimePeriodProfile
ScenePresentationProfile
CharacterAnimationProfile
WorldPickupDefinition
HUDLayoutProfile
UITheme
AudioAmbienceProfile

每個場景只需引用設定：

{
  "scene_id": "floor_6_hallway",
  "lighting_profile": "hallway_fluorescent",
  "time_profile": "indoor_apartment",
  "audio_profile": "floor_6_normal",
  "pickup_profile": "hallway_items",
  "hud_profile": "exploration_default"
