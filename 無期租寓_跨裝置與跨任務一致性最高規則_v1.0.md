# 《無期租寓》跨裝置與跨任務一致性最高規則 v1.0

最後更新：2026-07-16
規則層級：所有 Codex 任務、電腦、對話與開發者的最高進場、交接與完成判定規則。

## 1. 目的與核心原則

本規則確保不同電腦、不同 Codex 任務、不同對話與不同開發者維持相同品質、比例、風格、技術標準與驗收結果。A 電腦、B 電腦或任何新任務不得依聊天紀錄自行重建標準。

聊天紀錄不是專案；本機資料夾不是專案；某次生成結果不是專案。真正的專案是：

- 正式規格文件
- approved 基準素材
- 共用元件與 Design Token
- 正式資料表
- 驗收與回歸測試
- 最新 Git 提交

## 2. 來源優先級

1. 根目錄所有標示為「最高規則／最高規範」的文件，包括角色動畫、骰子動畫、2.5D Animation Bible、UI 美術、音訊系統與本文件。
2. 最新總交接文件 `無期租寓_開發總交接文件_v2.0.md`。
3. 專案內正式資料表、Design Token、元件庫與驗收腳本。
4. 最新程式碼與已提交素材。
5. 當前任務中使用者新下達的明確變更；必須先寫回正式規格，再開始實作。
6. 歷史聊天、舊對話、舊輸出、舊截圖與未提交草稿只供背景理解。

聊天與文件衝突時以正式文件為準。聊天出現新決定而文件尚未更新時，必須先更新規格，再修改程式或素材，並讓規格與實作進入同一版本。

## 3. 每次任務開始的固定流程

### 3.1 Git 核對

先執行並記錄：

```text
git status
git branch --show-current
git log -1
git remote -v
```

確認分支、遠端、最新提交與工作樹狀態清楚。工作樹有既有修改時不得覆蓋或強制同步。

### 3.2 同步

乾淨工作樹才能執行 `git pull --ff-only`。若工作樹不乾淨、沒有網路或沒有遠端權限，必須明確記錄阻礙與目前 commit，不得用 reset、checkout 或覆寫迴避。

### 3.3 固定必讀

至少完整閱讀：

- `AGENTS.md`
- 本文件
- `無期租寓_開發總交接文件_v2.0.md`
- `無期租寓_待辦追蹤_v1.0.md`
- `無期租寓_實測回饋與重改紀錄_v1.0.md`
- 與當前任務相關的所有最高規則

### 3.4 基準素材

必須從專案固定位置讀取，不得只看聊天圖片：

- `docs/references/approved/`
- `docs/references/REFERENCE_INDEX.md`
- `outputs/concepts/`
- `public/visual-benchmark.png`

### 3.5 任務前核對紀錄

大量修改前必須回報：

- 採用的 Git commit
- 採用的最高規則文件
- 採用的 approved 視覺基準
- 明確不修改的系統
- 本次驗收標準

## 4. 跨裝置一致環境

A／B 電腦必須使用相同 Git 分支、`package-lock.json`、Node 版本、建置與測試指令、字體、圖像基準、UI token、角色比例表、動畫與音訊規格、匯出尺寸及驗收腳本。

不得因螢幕、GPU、對話、使用者目錄或工作資料夾不同而更改風格、比例或完成標準。

## 5. 唯一視覺基準包

正式基準只存在於：

```text
docs/references/approved/
├── ui/
├── characters/
├── animation/
├── environments/
├── lighting/
├── audio/
└── rejected/
```

聊天圖片必須先複製至 approved、重新命名、加入 `REFERENCE_INDEX.md`、提交 Git 並由規格引用，才能成為正式基準。未進 approved 的圖片只視為臨時參考。

## 6. 基準索引

`docs/references/REFERENCE_INDEX.md` 每項至少記錄：檔名、用途、優先級、必須沿用、不可沿用、版本與取代關係。圖片內 AI 生成文字、房號與數值不得直接視為正式資料。

## 7. 共用 Design Token 與元件庫

所有 UI 必須引用 `app/design/tokens.ts` 的集中設定，不得在頁面自行硬編碼新顏色、字級、邊框、陰影、透明度、動畫時間或 z-index。

正式共用元件至少包括：

- `PaperPanel`
- `MetalFrame`
- `ArchiveCard`
- `ManagementButton`
- `ProgressHeader`
- `LeaseStamp`
- `AdministrativeDivider`
- `StatusLedger`
- `ObjectiveNote`
- `DialogNotice`

需要新增元件時，必須先說明現有元件不足、更新元件規格、補測試頁，再用於正式畫面。

## 8. 角色尺度與動畫驗收

所有角色必須使用 `app/game/characterScale.ts`，不得自行設定圖片高度。尺度以門洞為共同校準：標準成人為門高的 0.78–0.82，高瘦角色為 0.84–0.88；Boss 另依規格，但不得破壞地面透視。

動畫統一要求：雙腳中心 Pivot、固定 Ground Line、同角色幀尺寸一致、光影色溫與比例一致、無背景殘留、無棋盤格、無臉部漂移、無服裝變形、不得少量幀硬湊。

每套動畫需產生 contact sheet、silhouette overlay、ground-line check 與 brightness-difference check，並通過 Animation Validator。

## 9. 自動視覺回歸

固定驗證解析度：

- 1920×1080
- 1366×768
- 手機橫向

最低場景：首頁、入住登記、骰子結果、選擇建檔、身份確認、承租房、公共走廊、電梯、管理室、B1、B2、HUD、置物櫃。

每次修改後產生截圖並與 approved 基準比對；超過差異門檻必須列出差異，不得直接聲稱完成。

## 10. 禁止自行美化

沒有明確需求時，禁止改字體、加大圓角、漸層、霓虹、玻璃擬態、發光、更換紙張或按鈕、改角色比例、改 UI 左右架構、重新生成已定案素材。「我認為更好看」不是有效理由。

## 11. 生成素材固定流程

```text
確認既有素材
→ 閱讀相關最高規則
→ 使用 approved 基準
→ 中繼檔放 work/
→ 去背、裁切、校色、比例校正
→ 品質驗證
→ 移入 public/
→ 更新素材與授權紀錄
→ 提交 Git
```

禁止把第一次生成結果直接放進遊戲。

## 12. 任務結束前固定驗收

必須完成：`npm test`、正式 build、自動回歸、三種解析度截圖、視覺差異、素材引用、未追蹤檔案與 placeholder 檢查。

回報至少包含：完成項目、未完成項目、已知差異、測試結果、截圖位置、使用素材、修改文件與 Git commit。不得只說「已完成」。

## 13. A／B 電腦交接

完成端必須更新規格與待辦、執行測試、提交所有正式程式／素材／文件、推送 Git 並確認遠端提交存在。不得依賴本機未提交檔、聊天、`work/` 中繼檔或絕對路徑。

接手端必須 Pull、確認 commit、依 lockfile 安裝、執行測試並確認基準素材完整後才能修改。

## 14. 路徑規則

程式不得依賴 `C:\Users\某人\...` 等硬編碼使用者目錄。專案引用必須使用相對路徑。文件可以記錄工作路徑，但真正專案身份由 Git remote、branch、commit hash 與 `package-lock.json` 判定。

## 15. 衝突處理

發現文件、圖片、程式與聊天互相衝突時：

1. 列出衝突與各來源版本。
2. 依來源優先級選擇暫定來源。
3. 若最高規則彼此衝突，停止相關修改並詢問製作人。
4. 將決定寫回正式文件。

不得自行猜測或隱藏衝突。

## 16. 完成定義

只有程式功能、視覺、素材、測試、截圖驗收、文件、Git commit、Git push 與另一台電腦可重現全部達成，才能標記完成。缺少任何一項只能標記 `原型`、`待驗證`、`部分完成` 或 `TEMP`。
